Overview

What is this project?

This is a real-time industrial fire and smoke detection system built using Python, YOLOv8, and OpenCV. It monitors one or more camera feeds simultaneously, detects fire and smoke using a custom-trained deep learning model, and immediately triggers multi-channel alerts — sound, email, and a live web dashboard.

The system was built from scratch as a complete computer vision pipeline: from raw camera input, through AI inference, to alert delivery and session reporting. Every component was designed for reliability in low-resource environments — it runs entirely on CPU with no GPU required.

Features

What it can do

Every feature was built and tested end-to-end — nothing is placeholder or demo-only.

🔥

YOLOv8 Fire Detection

Custom-trained on the D-Fire dataset (20,320 images). Detects actual fire, not just orange colours. Ignores bird pictures, warm lights, and skin tones.

💨

YOLOv8 Smoke Detection

Uses the same trained model's smoke class. Replaced the original MOG2 colour-mask fallback that triggered on plant movement and body motion.

🌐

Live Web Dashboard

Flask server at localhost:5000. Shows live camera feed, confidence arcs, trend chart, detection zones, and scrolling alert log — all updating in real time.

🔔

Sound Alarm

Plays an alarm sound via pygame. Auto-generates a beep tone if no mp3 file is present. Respects cooldown to avoid repeated triggering.

📧

Email Alerts

Sends an HTML email with fire/smoke confidence stats and an attached screenshot of the detection frame. Sent in a background thread — never slows the video loop.

📋

Detection Log (CSV)

Every frame is written to a CSV file with timestamp, risk level, confidence scores, zone counts, and FPS. Alert events get a separate log with screenshot paths.

📊

Session HTML Report

Auto-generated when you quit. Shows session stats, confidence trend chart, risk distribution bars, critical screenshots, and the full alert log table.

📷

Multi-Camera Support

Monitors 2+ camera feeds simultaneously. Each camera runs in its own thread with independent detectors. Side-by-side view with combined risk assessment.

🛡

False Positive Filters

Scene-aware filter removes smoke/fire blobs overlapping YOLO-detected persons. Sustained alert gate requires 3 seconds of continuous detection before alerting.

⏱

Sustained Alert Gate

Prevents alert spam. A risk level must persist for 3 continuous seconds before sound/email alerts trigger. Single-frame spikes are ignored completely.

Pipeline

How it works — step by step

Every frame goes through this pipeline in under 200ms on a standard laptop CPU.

01

Camera Input

OpenCV opens the camera source — webcam index, video file path, or RTSP stream URL. Each frame is resized to 1280×720 for consistent processing.

02

Fire Detection — YOLOv8

The frame is passed to the custom-trained YOLOv8 model (fire_yolo.pt). The model returns bounding boxes with confidence scores for fire and smoke classes. Runs every 2nd frame to save CPU.

03

Smoke Detection — YOLOv8

The same model is reused for smoke detection, filtering only the smoke class. Falls back to MOG2 background subtraction + HSV colour masking if the model has no smoke class.

04

Scene Context — YOLOv8n

A second lightweight model (yolov8n.pt) detects persons, cars, buses, and trucks. Any fire/smoke blob that overlaps ≥35% with a person box is discarded — person movement ≠ smoke.

05

Risk Classification

Fire and smoke confidences are combined into a single risk score. Thresholds map to CLEAR → CAUTION → WARNING → CRITICAL. A sustained gate requires 3 continuous seconds before escalating.

06

HUD Rendering

Bounding boxes with glow outlines, confidence bars, animated danger banners, top/bottom status bars, and cinematic scanlines are drawn onto the frame using OpenCV drawing functions.

07

Alert Delivery

On risk escalation: alarm sound plays, email sends with screenshot attached, dashboard updates, CSV row is written. All alert systems run in background threads — the video loop is never blocked.

Detection Engine

The AI model

The fire detection model was trained on the D-Fire dataset — 20,320 images of real industrial fire and smoke scenarios. The model uses the Roboflow 3.0 (YOLOv8-compatible) architecture trained for 20 epochs on a Tesla T4 GPU.

Alert System

Multi-channel alerts

When a hazard is confirmed, the system triggers up to four simultaneous alert channels — all in background threads so detection never pauses.

🔔

Sound Alarm

Plays alarm.mp3 via pygame. If the file is missing, a 880Hz sine-wave beep is auto-generated. 15-second cooldown prevents repeat triggering. Stops instantly when risk clears.

📧

Email Alert

Sends a styled HTML email via Gmail SMTP showing risk level, fire/smoke confidence, zone counts, and frame number. Attaches the exact screenshot frame that triggered the alert.

🌐

Web Dashboard

Live feed streamed as MJPEG to localhost:5000. Dashboard shows confidence arcs, trend chart (last 60 seconds), detection zone counts, and a scrolling alert log.

📋

CSV Log + Report

Every alert is written to alert_log_*.csv with timestamp and screenshot path. On session end, a self-contained HTML report is auto-generated with charts and thumbnails.

Usage

Running the system

bash

python app.py 0

# Video file
python app.py fire_test.mp4

# RTSP stream
python app.py rtsp://192.168.1.100:554/stream

# Skip person/vehicle detection (faster on slow CPU)
python app.py 0 --no-yolo

# Custom output file

Key technical decisions

DecisionWhy
YOLOv8 over HSV for fireHSV colour matching triggered on bird pictures, warm lights, and skin tones. YOLO learned from 20k real fire images and ignores these.
Shared model for smokeAvoids loading a second model into memory. The fire model's smoke class is reused by SmokeDetector, saving ~50MB RAM and load time.
Consecutive frame gateSingle-frame spikes from camera noise can't escalate risk. Smoke must appear in 8 consecutive frames before confidence rises.
Background threads for alertsEmail sending can take 2–5 seconds. Running it in a daemon thread means the video loop never drops frames during alert delivery.
Scene-aware blob filteringPerson movement + cream walls = false smoke. Discarding blobs that overlap person bounding boxes eliminated the most common false positive source.
